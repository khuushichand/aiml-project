# UX Review & Feature Exploration: Watchlists Template & Output Workflow

## Your Role

You are a senior UX engineer and product designer reviewing the template creation, editing, and output generation workflow of a content monitoring system called "Watchlists." Your goal is to produce a structured audit of current pain points followed by creative feature proposals — all grounded in the actual architecture described below.

## System Overview

Watchlists is a content monitoring and briefing platform. The workflow is:

1. **Sources** — Users add RSS feeds, websites, or forums to monitor
2. **Jobs** — Scheduled monitors that fetch from sources, apply filters (keyword/regex/author/date), and ingest matching items
3. **Runs** — Each job execution produces a run with captured items
4. **Templates** — Jinja2-based HTML or Markdown templates that define how a briefing looks
5. **Outputs** — A rendered briefing generated by applying a template to a run's items, then optionally delivered via email, chatbook, or ingested into a knowledge base

The frontend is organized into 8 tabs: Overview (dashboard + quick-setup wizard), Sources, Jobs, Runs, Items, Templates, Outputs, and Settings.

## Current Template Architecture

### Template Editor (`apps/packages/ui/src/components/Option/Watchlists/TemplatesTab/TemplateEditor.tsx`)
- **Modal-based editor** (antd `Modal`, width=800) with three tabs: **Editor**, **Preview**, **Variables**
- **Form fields**: name (slug, `[A-Za-z0-9_-]{1,64}`, disabled when editing), description (optional), format (Radio: `html` / `md`), content (raw `Input.TextArea`, 18 rows, monospace, no resize)
- **Quick-insert snippets** (append-only — always appends to end of content, does not insert at cursor position):
  - "Items loop": `{% for item in items %}\n{{ item.title }}\n{% endfor %}`
  - "Filter tallies": `{% if filter_tallies %}\n{{ filter_tallies }}\n{% endif %}` — **Note: `filter_tallies` is not actually injected into the template context by the backend (see Known Issues below)**
  - "Summary block": `{% if item.summary %}\n{{ item.summary }}\n{% endif %}`
  - "Generated timestamp": `{{ generated_at }}`
- **Preview tab**: Renders content as sanitized HTML via `marked` + `DOMPurify`; shows a warning that Jinja2 logic is **not** evaluated (static markup preview only)
- **Variables tab** ("Docs"): Inline reference listing available variables + a static `SAMPLE_TEMPLATE_CONTEXT_PAYLOAD` JSON example
- **Version management** (edit mode only): version selector dropdown, "Load version" button, "Load latest" button, unsaved-drift alert when content differs from loaded version baseline
- **No template validation endpoint** exists — there is no server-side Jinja2 syntax check before saving

### Template Storage (`tldw_Server_API/app/core/Watchlists/template_store.py`)
- **File-based**: `Config_Files/templates/watchlists/<name>.md` or `<name>.html`
- **Metadata** in `<name>.meta.json` (description, current_version, history array of older snapshots)
- **Name validation**: Must match `^[A-Za-z0-9_\-]{1,64}$`
- **Version history**: On every overwrite save, current content is pushed onto the `history` array in `.meta.json` and `current_version` is incremented
- **4 built-in templates** (seeded on first run via `_seed_defaults()` — only written if file doesn't exist):

| Name | Format | Description |
|---|---|---|
| `briefing_markdown` | md | Default markdown briefing — numbered items with title link, truncated summary (300 chars), tags |
| `newsletter_markdown` | md | Headline + article links + summary + published date per item |
| `mece_markdown` | md | Groups items by first tag into MECE categories using Jinja2 namespace trick |
| `newsletter_html` | html | Full HTML document with inline CSS styling; numbered items |

### Known Issues: Frontend-Backend Variable Mismatch
The Variables ("Docs") tab in TemplateEditor documents several variables that **do not exist** in the actual template context:
- `item.content` — not in context (use `item.summary` instead)
- `item.author` — not in context
- `item.source` — not in context (use `item.source_id`)
- `item.filter_matches` — not in context
- `filter_tallies` — not in context (a quick-insert snippet references it, but it would silently evaluate as falsy)

Conversely, several **real** context variables are **not documented** in the Variables tab:
- `{{ title }}`, `{{ items_markdown }}`, `{{ items_html }}`, `{{ item_count }}`, `{{ has_llm_summaries }}`, `{{ template_name }}`, `{{ template_description }}`, per-item `llm_summary`, `index`, `markdown_line`, `html_entry`

Additionally, the Python-side built-in template string constants in `template_store.py` use `{{ job_name }}` and `{{ run_id }}` which are **not** in the context dict (should be `{{ job.name }}` and `{{ run.id }}`). The on-disk template files have been corrected to use the right variables, but the Python constants remain stale.

### Output Generation (`POST /watchlists/outputs`)
**Request body** (`WatchlistOutputCreateRequest`) supports:
- `run_id` (required) — which run's items to use
- `item_ids` (optional) — subset of items; defaults to all `status="ingested"` items (up to 1000)
- `title` (optional) — defaults to `{job_name}-Output-{version}`
- `template_name` (optional) — name of a watchlists template
- `template_version` (optional) — pinned version number
- `format` — `"md"` or `"html"`; falls back to template format or job default
- `summarize` (bool) — triggers LLM summarization of items
- `llm_provider`, `llm_model`, `summarize_prompt` — LLM config for summarization
- `generate_tts` (bool) — triggers TTS audio briefing generation
- TTS config: `tts_template_name`, `tts_model`, `tts_voice`, `tts_speed`
- Audio briefing config: `generate_audio`, `target_audio_minutes`, `audio_model`, `audio_voice`, `audio_speed`, `audio_language`, `audio_llm_provider`, `audio_llm_model`
- Background audio mixing: `background_audio_uri`, `background_volume`, `background_delay_ms`, `background_fade_seconds`
- Persona-aware summarization: `persona_summarize`, `persona_id`, `persona_provider`, `persona_model`, `voice_map`
- Delivery: `deliveries` (email/chatbook overrides), `ingest_to_media_db` (bool)
- Lifecycle: `retention_seconds`, `temporary` (bool), `metadata` (arbitrary dict)

**Pipeline**: Look up run + job → resolve effective settings (merge job `output_prefs_json` with request) → load items → optionally run LLM summarization → build context via `_build_output_context()` → render Jinja2 template → write file to disk → create artifact row → optionally deliver via email/chatbook → return `WatchlistOutput`

## Part 1: Structured UX Audit

For each issue, provide: **Problem** (what's wrong), **Impact** (who it affects and how), **Severity** (Critical / High / Medium / Low), and **Recommendation** (concrete fix).

Evaluate the following areas:

### 1.1 Template Creation & Editing Flow
- How discoverable is template creation for a new user? (Consider: it's hidden behind the Templates tab, separate from the Jobs tab where outputs are configured)
- Is the editor adequate for the complexity of Jinja2 templates? (Currently a plain `Input.TextArea` with no syntax highlighting, no auto-complete, no bracket matching)
- How well does variable insertion work? (Quick-insert snippets append to end only — no cursor-position insertion; `insertSnippet()` always appends)
- Is the preview feedback loop fast enough? (Edit tab → Preview tab → back to Edit tab; preview does NOT evaluate Jinja2 — it only renders static markup)
- How clear is the relationship between templates and jobs/outputs? (Templates tab is 2 tabs away from Jobs tab)
- Version management UX — is it clear how to view/restore old versions? (Version selector dropdown + Load button; unsaved-drift alert exists)
- **Critical**: The Variables tab documents variables that don't exist in the backend context (`item.content`, `item.author`, `item.source`, `item.filter_matches`, `filter_tallies`) and omits real variables (`title`, `items_markdown`, `items_html`, `item_count`, `has_llm_summaries`, etc.) — see Known Issues above. Evaluate the impact of this mismatch on template authoring.
- **Missing**: No server-side template validation endpoint exists. Users can save syntactically invalid Jinja2 that will fail silently at render time.

### 1.2 Template-to-Output Pipeline
- How intuitive is the flow from "I have a run" to "I have a rendered briefing"? (User must navigate Runs tab → note run ID → go to Outputs tab → generate)
- Is it clear which template will be used when generating an output? (Job has `output_prefs` defaults; output generation can override)
- How does the user understand what variables are available? (The Variables tab in TemplateEditor is inaccurate — see Known Issues)
- Error feedback when template rendering fails — is it actionable? (What happens when a template references `{{ filter_tallies }}` which doesn't exist?)

### 1.3 Output Review & Iteration
- Can the user easily iterate on a template by seeing how it renders against real data? (Preview tab shows static markup only, not rendered against actual run data)
- Is there a way to preview a template against a specific run *before* generating an output?
- How does the user compare different template versions or different templates side-by-side?

### 1.4 Cross-Tab Navigation
- How well connected are the 8 tabs (Overview, Sources, Jobs, Runs, Items, Templates, Outputs, Settings)?
- Can the user navigate from an output back to its template to edit it?
- Does the job creation form's "Output & Delivery" section make it clear how templates relate to scheduled outputs?
- The guided presets in JobFormModal (`briefing_md`, `newsletter_html`, `mece_md`) — are these discoverable? Do they map clearly to the built-in templates?

## Part 2: Feature Exploration — LLM-Powered Briefings

Explore these feature ideas in depth. For each, describe: **Concept**, **User Story**, **UX Wireframe** (text-based), **Data Flow** (what calls what), **Edge Cases**, and **Implementation Complexity** (Low/Medium/High).

### 2.1 LLM-Generated Briefing Introduction/Summary
The user wants an AI-written introduction at the top of each briefing that summarizes all captured items into a cohesive narrative overview — like an executive summary or editorial intro for a newsletter.

Consider:
- Where in the pipeline should the `{{ briefing_summary }}` variable be generated? The output endpoint already supports `summarize` (bool) + `llm_provider` / `llm_model` / `summarize_prompt` for per-item LLM summaries (populating `item.llm_summary`). Should the briefing-level summary be a separate pipeline step or an extension of the existing summarization?
- Should this be generated at render time (synchronous LLM call during `POST /watchlists/outputs`) or as a pre-computation step (e.g., triggered after a run completes)?
- How does the user configure the LLM provider, model, and tone/style for the briefing summary vs. per-item summaries?
- What happens if the LLM call fails — graceful fallback? The template should degrade gracefully: `{% if briefing_summary %}{{ briefing_summary }}{% else %}No summary available.{% endif %}`
- Should the user be able to edit the generated summary before the output is finalized? (Currently, outputs are generated and stored atomically — there is no draft/review step)
- Token budget: if there are 200 items with summaries, how do you fit them into an LLM context? Options: hierarchical summarization (summarize groups, then summarize summaries), truncation, or map-reduce
- Cost/latency implications: the output endpoint is synchronous — adding a large LLM call could make it very slow

### 2.2 Result Grouping with Group Summaries
The user wants to group items within a briefing (e.g., by topic, source, tag, or custom category) and have each group preceded by a short AI-generated summary of that group's items.

Consider:
- Grouping strategies: by tag, by source, by LLM-classified topic, by user-defined rules, by date range
- The `mece_markdown` built-in template already does basic tag-based grouping in pure Jinja2 (using a namespace trick to group by first tag) — how should this evolve?
- Should grouping be a template concern (Jinja2 logic) or a backend concern (pre-grouped context providing `{{ groups }}` as a list of `{name, items, summary}` dicts)?
- Per-group LLM summaries: same token budget / latency / failure questions as 2.1 but multiplied by number of groups
- UX for defining group rules: drag-and-drop? filter builder? tag picker? Or should this be purely template-driven with documentation?
- What if an item belongs to multiple groups (multiple tags)? Duplication vs. primary-group assignment
- How does the user reorder groups within the briefing?
- Should groups be defined at the job level (in `output_prefs`) or at the template level or at output-generation time?

### 2.3 Template Customization Power-Ups
Explore additional customization capabilities:
- **Syntax-highlighted editor**: Replace `Input.TextArea` with CodeMirror or Monaco editor with Jinja2 + HTML/Markdown syntax highlighting
- **Cursor-position snippet insertion**: Fix `insertSnippet()` to insert at cursor position instead of appending to end
- **Server-side validation**: Add a `POST /watchlists/templates/{name}/validate` endpoint that checks Jinja2 syntax before saving
- **Live preview against real data**: Allow selecting a run and rendering the template against its actual items (server-side Jinja2 evaluation, not just static markup)
- **Conditional sections**: Show/hide template blocks based on item count, tag presence, `has_llm_summaries`, etc.
- **Custom CSS/styling**: Per-template styling for HTML outputs
- **Template inheritance**: Base templates with override blocks (Jinja2 natively supports `{% extends %}` and `{% block %}`)
- **Template marketplace/sharing**: Import/export templates between users
- **Drag-and-drop template builder**: Visual block-based editor as alternative to raw Jinja2 for non-technical users
- **Smart defaults**: Auto-suggest template structure based on job type (news digest vs. research report vs. competitive intelligence)
- **Fix the Variables tab**: Synchronize the documented variables with what `_build_output_context()` actually provides

### 2.4 Output Polish & Delivery
- **Inline editing**: Edit the rendered output before delivery (fix LLM hallucinations, add commentary) — currently outputs are generated atomically with no draft step
- **Approval workflow**: Review → approve → deliver pipeline for team use
- **Multi-format export**: Same content rendered as HTML email, PDF, Slack message, Markdown doc (the backend already supports `md` and `html` formats; extend to more)
- **Scheduled delivery**: Auto-generate and deliver outputs on a schedule tied to job runs (the job form already has email/chatbook delivery config, but is this actually wired to auto-trigger on run completion?)
- **Diff view**: Compare this output to the previous one to see what's new (version incrementing per run already exists)
- **Template A/B testing**: Generate the same run's output with two different templates side-by-side to compare
- **TTS/Audio preview**: The output endpoint supports `generate_tts` and audio briefing with background music mixing — is the UX for configuring and previewing audio outputs adequate?
